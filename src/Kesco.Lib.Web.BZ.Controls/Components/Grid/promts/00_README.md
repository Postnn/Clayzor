# Настраиваемый фильтр KescoGrid — мастер-план (порядок выполнения)

Цель: рекурсивный конструктор фильтров (query-builder в стиле Telerik Filter)
поверх существующего грида, с переиспользованием его типов (`ColumnType`,
`ColumnFilterOperator`, `LogicalOperator`, `ColumnFilter`, диалоги через `IDialogService`).

## Статус
- **01 — расширение типов (Date/Decimal/NULL) — СДЕЛАНО, закоммичено.**
  (Его switch-подход к типам далее рефакторится в задаче 03.)
- **02 — рекурсивные типы фильтра — СДЕЛАНО, закоммичено.**
- **03 — типобезопасная модель колонок (дескрипторы) — СДЕЛАНО, закоммичено.**
- **04 — вынос @code в KescoGrid.razor.cs — СДЕЛАНО, закоммичено.**
- **05 — разбиение code-behind по темам — СДЕЛАНО, закоммичено.**

затем **новые компоненты** (07–09), затем **сборка всего вместе** (10–11),
после — опции и тесты.

## Правила работы (каждую сессию)
1. Бери **одну** задачу. В начале дай агенту этот файл (00) + текущий промпт.
   Скажи: «сделай только задачу N, потом остановись».
2. Перед кодом — пусть покажет план изменений (что и где правит).
3. Рефакторинги (03–06) — **behavior-preserving**: поведение не меняем, только
   раскладку/абстракцию. Перед каждым — git-коммит (точка отката).
4. Конец задачи = `dotnet build` без ошибок + чек-лист промпта + коммит.
5. Следующую задачу — желательно в новой сессии (чистый контекст).
6. Если агент без доступа к репозиторию — вместе с промптом вставляй реальные
   файлы, на которые он ссылается (иначе выдумает API).

## Соглашения проекта (повторять агенту)
- Переиспользуем существующие типы; новые — только где их нет.
- Каждый компонент: `.razor` + `.razor.cs` (логика не в разметке).
- Диалоги — через `IDialogService` + `IMudDialogInstance` (как `KescoColumnFilterDialog`),
  не через `@bind-Visible`.
- Значения фильтра в SQL — только Dapper-параметрами; имя колонки — только из реестра колонок.
- partial-классы повторяют точную сигнатуру дженерика; база/интерфейсы — в одном файле.
