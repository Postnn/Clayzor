## Контекст

- Решение: `Clayzor.sln`. Стек: **.NET 10 Blazor Server (Interactive Server)**, MudBlazor 9.x, Dapper + `Microsoft.Data.SqlClient`, SQL Server 2008 R2.
- Приложение-потребитель: `Kesco.App.Web.Inventory` (точка входа). **Сначала подтверди по `Clayzor.sln` фактическое имя веб-проекта** — код фичи туда **не** пишем, приложение только подключает готовый компонент.
- Вся реализация — в общей библиотеке **`Clayzor.Lib.Web.Controls`** (Razor Class Library). В приложении допустима только регистрация сервиса в `Program.cs` и одна строка в `MainLayout`.

## Проблема (из реального стектрейса)

При недоступном SQL Server (`SqlException`, error 40 / provider Named Pipes / «Сервер не найден или недоступен») исключение **всплывает как необработанное** во время инициализации компонента:

```
DbManager.get_Connection()  -> SqlConnection.Open()  -> SqlException
  -> DbManager.RunAsync<T>()
  -> DynamicSql.QueryRowsAsync()
  -> ClayGridDefinitionData.LoadGridAsync()
  -> ClayGrid.InitDynamicMode() / OnInitializedAsync()
  -> ... -> DeveloperExceptionPageMiddleware  (страница падает)
```

То есть пользователь видит страницу ошибки, а не аккуратное сообщение.

## Чего хотим (требования заказчика)

1. При недоступности БД пользователь видит **аккуратный оверлей** (по поведению — как штатный экран «Соединение потеряно» в Blazor, но это **наша** реализация на уровне БД, см. предупреждение ниже).
2. Приложение **само пытается переподключиться**: **3 попытки, интервал 30 секунд** между попытками.
3. Если восстановить соединение **не удалось** — показать **статичное сообщение** и **кнопку «Перезагрузить страницу»**.
4. Если соединение восстановилось — оверлей исчезает, данные страницы подгружаются заново (страница снова работоспособна без ручной перезагрузки).

## ⚠️ Главное, что нельзя перепутать

- Обрыв **SignalR-цепочки Blazor (circuit)** и **недоступность SQL Server** — это **разные** вещи. В нашем случае circuit жив, отвалилась именно БД.
- **Запрещено** трогать штатный reconnection Blazor: `Blazor.start({ reconnectionOptions })`, разметку `#components-reconnect-modal`, `App.razor`-скрипты reconnect. Мы делаем **похожий по виду и логике** оверлей, но реагирующий на **ошибки БД**, а не на разрыв circuit.
- Числа 3 и 30 сек — про **переподключение к БД**, а не про SignalR.

## Шаг 1 — Классификация «ошибка недоступности БД»

Добавить в DALC/сервис ошибок метод-классификатор, отличающий **потерю связи с сервером** от обычных ошибок запроса (синтаксис, нарушение ключа и т.п.). Реагировать оверлеем **только** на потерю связи.

- Ориентир — коды `SqlException`: сетевые/недоступность сервера (например `2`, `53`, `40`, `-2`, `258`, `11001`, `1231`, `4060` и подобные) и `Win32Exception (2)` «Не удаётся найти указанный файл» внутри. Точный набор — на твоё усмотрение, но **держи список в одном месте** и задокументируй `///`-комментарием, почему именно эти коды.
- Не изобретай сложную иерархию — один статический метод `bool IsConnectivityError(SqlException ex)`.

**→ verify:** unit-тест в `tests/Clayzor.Lib.Web.Controls.Tests` (или соответствующем тестовом проекте): «ошибка связи → true», «ошибка запроса → false». Проверь `dotnet test`.

## Шаг 2 — Не ронять страницу: гасить связь-ошибку в choke-point

Сделать так, чтобы **связь-ошибка не всплывала как необработанное исключение**. Правка **точечная** — в `DbManager.RunAsync<T>()` (и там, где открывается соединение):

- при `IsConnectivityError` — уведомить `ISqlErrorHandler` (как сейчас) и **не пробрасывать** исключение наверх, а вернуть безопасное значение по умолчанию (пустой результат/`default`), чтобы компонент дорендерился «пустым» под оверлеем;
- **обычные** (не связь-) ошибки — оставить **прежнее поведение** (не менять!).

Не рефактори соседний код, не «улучшай» то, что не просили (Surgical Changes).

**→ verify:** при недоступной БД инициализация `ClayGrid` больше **не** приводит к `DeveloperExceptionPageMiddleware`; страница рендерится, а сверху появляется оверлей (см. Шаг 3).

## Шаг 3 — UI-оверлей `ClayDbReconnectOverlay` в `Clayzor.Lib.Web.Controls`

Новый компонент в библиотеке (например `Components/Connectivity/ClayDbReconnectOverlay.razor` — путь согласуй со структурой репозитория). Требования:

- Подписывается на событие «ошибка связи БД» из `ClayErrorService`/`ISqlErrorHandler` (обновление UI — через `InvokeAsync(StateHasChanged)`).
- Пока идёт восстановление — модальный оверлей (MudBlazor: `MudOverlay`/`MudProgressCircular`) с текстом вида **«Потеряно соединение с базой данных. Попытка переподключения… (N из 3)»**. Диалоги/оверлеи проекта — draggable там, где это диалоги; здесь это блокирующий оверлей.
- Текст — **русский**. Стиль — фирменные цвета/типографика проекта (см. `STYLE_RULES.md`), переиспользуй существующие компоненты (`ClayButton` для кнопок с тултипом и т.п.).

**→ verify:** оверлей появляется при связь-ошибке и скрыт в норме.

## Шаг 4 — Логика авто-переподключения (3 × 30 сек)

Внутри оверлея (или в отдельном scoped-сервисе, если так чище — обоснуй выбор):

- при появлении связь-ошибки запустить цикл: **до 3 попыток**, **пауза 30 секунд** между попытками (`PeriodicTimer`/`Task.Delay` + `CancellationToken`);
- каждая попытка — **лёгкий health-check через существующий `DbManager`** (тривиальный запрос, например `SELECT 1`), а **не** повтор тяжёлого запроса страницы;
- **успех** → остановить цикл, скрыть оверлей и **перезагрузить данные текущей страницы** (варианты: мягкий `NavigationManager.NavigateTo(currentUri, forceLoad: false)` для повторного прогона загрузки, либо событие «reconnected», на которое подписаны страницы — выбери один способ и обоснуй);
- **все 3 попытки неуспешны** → перейти в терминальное состояние (Шаг 5).
- Числа `3` и `30 сек` вынести в **именованные константы** с `///`-комментарием (не «магические» числа), но **без** избыточной конфигурируемости, которую не просили (Simplicity First).

**→ verify:** в логах/на экране виден счётчик попыток «1 из 3 → 2 из 3 → 3 из 3» с интервалом ~30 сек.

## Шаг 5 — Терминальное состояние: статичное сообщение + кнопка

После 3 неудачных попыток:

- заменить содержимое оверлея на **статичное сообщение** (например «Не удалось восстановить соединение с базой данных.») и кнопку **«Перезагрузить страницу»**;
- кнопка выполняет полную перезагрузку: `NavigationManager.NavigateTo(NavigationManager.Uri, forceLoad: true)` (или JS `location.reload()`), без дальнейших авто-попыток.

**→ verify:** после исчерпания попыток авто-ретраи прекращаются, показано статичное сообщение, кнопка перезагружает страницу.

## Шаг 6 — Подключение в приложении (минимум кода)

- В `Program.cs` приложения — при необходимости зарегистрировать сервис оверлея (если сделал отдельный сервис). Реюзать уже зарегистрированный `ClayErrorService`/`ISqlErrorHandler`.
- В `MainLayout` приложения — добавить **одну** строку с компонентом `<ClayDbReconnectOverlay />` (рядом с `ClayErrorBar`).
- Больше **ничего** в приложении не менять.

**→ verify:** приложение (`Kesco.App.Web.Inventory` / подтверждённое имя) собирается и запускается; фича работает, будучи целиком в библиотеке.

## Ручной тест (обязателен перед сдачей)

1. Запусти приложение с рабочей БД — открой страницу с `ClayGrid` (динамический режим). Всё работает как раньше.
2. **Оборви БД**: останови службу SQL Server **или** временно подставь заведомо неверный сервер в строке подключения (`appsettings.Development.json`, он в gitignore).
3. Обнови/открой страницу с гридом:
   - страница **не** падает в Developer Exception Page;
   - появляется оверлей с переподключением, счётчик «N из 3», интервал ~30 сек;
4. **Верни БД** до исчерпания попыток → оверлей исчезает, данные грузятся, ручная перезагрузка не нужна.
5. Повтори, **не возвращая** БД → после 3 попыток статичное сообщение + рабочая кнопка «Перезагрузить страницу».

## Критерии приёмки (Definition of Done)

- [ ] Связь-ошибка БД **не** приводит к необработанному исключению / странице ошибки.
- [ ] Показывается аккуратный оверлей на **русском**, стиль соответствует проекту.
- [ ] Ровно **3** попытки с интервалом **30 сек**, `SELECT 1`-health-check через `DbManager`.
- [ ] Успех → авто-восстановление без ручной перезагрузки. Провал → статичное сообщение + кнопка перезагрузки.
- [ ] Обычные (не связь-) ошибки БД ведут себя **как раньше** (regression не допущен).
- [ ] Вся логика — в `Clayzor.Lib.Web.Controls`; в приложении только регистрация + одна строка в `MainLayout`.
- [ ] Не тронуты reconnection-настройки Blazor/SignalR и `#components-reconnect-modal`.
- [ ] `dotnet build Clayzor.sln` и `dotnet test` — зелёные. Новые публичные типы/члены имеют `/// <summary>`.
- [ ] Диф минимален: каждая изменённая строка прослеживается к требованию. Git-коммит **не** делать без отдельного указания.

## Вне рамок (не делать)

- Настройка/замена штатного reconnection SignalR-цепочки Blazor.
- Ретраи на уровне отдельных запросов Dapper / политики Polly для всех вызовов.
- Конфигурируемость числа попыток/интервала через UI или сложные настройки (только именованные константы).
- Рефакторинг `DbManager`/`ClayErrorService` сверх точечной правки классификации и гашения связь-ошибки.
